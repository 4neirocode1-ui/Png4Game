# Архитектура Png4Game (конвейер обработки иконок)

**Последнее обновление:** 2026-06-11

Проект-«мастерская»: берёт лист с иконками (сгенерированный нейросетью Nano Banana),
убирает фон в прозрачность, нарезает на отдельные иконки, приводит к единому квадратному
размеру (по умолч. 256×256) и сохраняет в WebP (или PNG) для игры. Сама игра — в соседнем проекте
`Documents/Game` (рогалик на Godot). Арт-направление и правила генерации листов —
в [Описание.txt](Описание.txt); смысл/имена иконок — в `Game/scripts/icon_registry.gd`.

---

## Правила работы (грабли, на которых уже стояли)

1. **Кодировка / `.bat`.** НЕ класть кириллицу внутрь `.bat` — cmd парсит файл в
   системной локали (cp1251/cp866), кириллица превращается в кракозябры. Решение:
   `.bat` держим пустым ASCII-стартером, ВЕСЬ русский текст и меню — в Python
   (`sys.stdout.reconfigure(encoding="utf-8")` в начале скрипта надёжно держит UTF-8).
   См. `launcher.py` + `Иконки.bat`. То же правило записано в `Game/plans/architecture.md §1`.

2. **Правки, меняющие ВИД иконок, — только с одобрения пользователя.** Тестирует и
   оценивает результат пользователь (он креативный директор). AI показывает «до/после»
   на пробе и ждёт решения, а НЕ зашивает изменение в код по своей оценке.
   Записано после случая 2026-06-11 (дефриндж-эрозия — см. «Испробовано и отклонено»).

3. **Git есть — пользуйся им для отката.** Репозиторий локальный (с 2026-06-11,
   ветка `main`, без удалёнки). Схема: довели результат до состояния, которое
   нравится → коммит (`git add -A && git commit -m "..."`). Дальше экспериментируем
   спокойно — неудачный заход откатывается одной командой `git restore .` (или
   `git restore <файл>`), без ручного копирования. `git diff` — посмотреть, что
   изменилось с последнего коммита. В `.gitignore`: `.venv`, `__pycache__`, `.idea`,
   `output/` (генерируется), `User_files/` (личные пробы), `settings.json` (локальные настройки).

4. **Текст на листе убирать в ИСТОЧНИКЕ, а не кодом.** Подписи под предметами
   ломают и нарезку (ловятся как иконки), и нумерацию (сдвиг индексов). Правильно —
   генерировать листы без подписей (это требует и `Описание.txt`). Бороться кодом
   с текстом, который можно просто не рисовать, — лишний костыль.

## Запуск

| Файл | Назначение |
|---|---|
| `Иконки.bat` | Двойной клик из Проводника. ASCII-стартер: проверяет `.venv`, запускает `launcher.py`. |
| `launcher.py` | Меню (UTF-8). При запуске выбирает исходник из `input/` (если один — берёт сам). Результаты → `output/<имя_исходника>/`. [1] нарезать, [2] срез внешнего, [3] срез внутреннего, [4] резкость, [5] анализ, [6] настройки (размер/формат), [7] сменить исходник, [8] открыть папку, [0] выход. Настройки помнятся в `settings.json`. |

## Скрипты конвейера

| Файл | Назначение |
|---|---|
| `run.py` | Точка входа нарезки: проверяет структуру папок и `input/1.png`, вызывает `process_opencv.process()`. |
| `process_opencv.py` | **Основной путь.** Сам определяет тип листа: прозрачный фон (RGBA) → режет по альфе; белый фон → старый путь. Детект → почистка → центровка на заданный размер. `process(input_file, output_dir, target_size, fmt)`. |
| `utils.py` | Ядро: загрузка, построение альфы, детект иконок (по силуэту и по альфе), чистка, `trim_edges` (срез внешнего/внутреннего края), `list_icon_files` (png/webp), вписывание в холст, сохранение в выбранном формате. Все НАСТРОЙКИ — в шапке файла. |
| `process_rembg.py` | Альтернативный путь через нейросеть-вырезатель rembg. **СЛОМАН:** импортирует из `utils` несуществующую `find_icon_bboxes` (есть `find_icons`/`find_icons_by_alpha`). Мёртвый путь, пока не починить. |
| `enhance.py` | Постобработка резкости (unsharp mask), пресеты `soft`/`hard`, интерактивный режим `review`. Источник: `input/enhance/` или `output/opencv/`. |
| `analyze.py` | **Анализатор результата.** Сканирует `output/opencv/`, ищет проблемы (бахрома снаружи/внутри, чисто-белые пятна, дыры) и печатает отчёт. На его основе планируется авто-проход нужных инструментов. Ничего не меняет — только смотрит. |
| `prompts.py` | Русские строки для `enhance.py` (вынесены отдельно, чтобы спелл-чекер PyCharm не молотил по ним). |
| `Описание.txt` | Арт-спек: правила вида иконок + правила раскладки листа для авто-нарезки. |

### Папки
В `input/` — НЕСКОЛЬКО листов со своими именами; на каждый своя папка результата
`output/<имя>/`. (Имя = файл без расширения: `weapons.png` → `output/weapons/`.)
```
input/
  weapons.png        # листы на нарезку (любые имена, .png/.webp)
  armor.png
  enhance/           # (опц.) выборочные иконки для enhance.py (подпапка, не источник)
output/
  weapons/           # всё по исходнику weapons.png:
    opencv/          #   результат нарезки: icon_NNN.png + icon_NNN.webp
    trimmed/         #   (опц.) срез внешнего контура 1px (меню [2])
    trimmed_inner/   #   (опц.) срез внутренних кромок 1px (меню [3])
    enhanced/        #   (опц.) резкость enhance.py: <preset>/icon_NNN.png
    analyze_report.json   # отчёт analyze.py (машиночитаемый)
User_files/          # личные сравнения/пробы пользователя (в .gitignore)
```
`run.py` — legacy-вход (нарезка дефолтного `input/1.png`); основной путь —
через `launcher.py` с выбором исходника.

## Поток основного пути (process_opencv)

`process()` читает лист и `_has_transparent_bg()` выбирает путь.

**Путь А — прозрачный фон (RGBA, наш текущий рабочий процесс).** Пользователь
убирает фон в редакторе → отдаёт чистую альфу. Скрипт её УВАЖАЕТ, не пересчитывает.
```
input/1.png (фон уже прозрачный)
  → find_icons_by_alpha   (детект иконок по альфе)
  на каждую иконку:
    → берём исходную альфу как есть, зануляем чужие компоненты в bbox
    → keep_main_component  (убирает прилипший мусор)
    → fill_small_holes     (спеклы антиалиасинга < MAX_HOLE_AREA px)
    → fit_to_canvas        (premultiplied-alpha ресайз + центровка на TARGET_SIZE)
  → save_icons            (формат по настройке: webp / png / both)
```
Тут ни белой заливки, ни пробивки окон: что прозрачно в исходнике (фон,
почищенная полость лука) — то прозрачно в результате.

**Путь Б — сырой белый фон (запасной, лист прямо из Nano Banana).**
```
  → load_image            (RGB под alpha=0 заливаем белым — гасит скрытый текст из PSD)
  → build_global_alpha    (flood fill белого фона от краёв + мягкий 2px-край)
  → find_icons            (силуэты по gray-порогу; подписи отсекаются по форме aspect)
  → keep_main_component → fill_small_holes → fit_to_canvas → save_icons
```
`punch_internal_white` в обоих путях ОТКЛЮЧЁН (см. ниже).

**Опционально, отдельными шагами (`trim_edges`):**
- **[2] срез внешнего контура** — безопасно, для всех иконок (убирает светлую кайму).
  → `output/<имя>/trimmed/`.
- **[3] срез внутренних кромок** (вокруг полостей) — полезен для колец/петель, но
  ТОНЧИТ тонкие внутренние линии (тетива лука), поэтому применять осознанно.
  → `output/<имя>/trimmed_inner/`.

Оба читают `output/<имя>/opencv/`, оригинал нарезки не портят.

## Ключевые настройки (`utils.py`, шапка)

| Константа | Знач. | Смысл |
|---|---|---|
| `TARGET_SIZE` | 256 | Сторона квадрата итоговой иконки (256 иконки / 512 спрайты / 1024 портреты). Меняется в меню [6]. Иконка ВСЕГДА этого размера — числа в логе нарезки (415×469…) это bbox НА ЛИСТЕ, не результат. |
| `OUTPUT_FORMAT` | webp | Что сохранять: `webp` (по умолч., в 2-4× легче) / `png` / `both`. Меню [6]. |
| `PADDING` | 4 | Отступ от края холста. |
| `MIN_ICON_SIZE` / `MIN_ICON_AREA` | 35 / 500 | Отсекают мусорные мелкие компоненты. |
| `MAX_ASPECT_RATIO` | 3.0 | Отсекает длинные узкие подписи. |
| `MAX_HOLE_AREA` | 8 | Замкнутые дыры < N px заливаются (только спеклы антиалиасинга 1–4 px). Абсолютный порог, не %. |
| `BG_TOLERANCE` | 5 | Узкий допуск flood fill — не «протекает» сквозь тонкую обводку. |
| `ALPHA_FADE_LO/HI` | 200 / 250 | Диапазон смягчения края по яркости. |
| `DETECT_GRAY_HI` | 230 | Темнее — часть иконки при детекте. |

## Испробовано и отклонено

- **Дефриндж-эрозией (срезать 1px контура от светлой каймы).** 2026-06-11. Убирала
  бахрому белого антиалиасинга на тёмном UI, НО подтачивала кромки лезвий — «съедала»
  оружие. Размен плохой: бахрома мешает меньше, чем потерянный металл. **Откатили.**
  Если возвращаться к бахроме — пробовать не эрозию, а деконтаминацию цвета края
  (тянуть цвет из внутреннего пикселя), не трогая силуэт.

- **Лист С подписями + отбраковка текста кодом.** Подписи ловятся как иконки и
  сдвигают нумерацию. Отказались в пользу безтекстовых листов (правило 4 выше).

- **`punch_internal_white` (авто-пробивка белых окон внутри иконки).** 2026-06-11.
  По цвету не отличает «окно к фону» (петля арбалета) от «яркого блика стали» →
  сверлила сквозные дыры в лезвиях мечей/кинжалов. **Отключена** (код в `utils.py`
  оставлен, но не вызывается). Замкнутые окна (петля арбалета, дырки колец) теперь
  убираются вручную в редакторе — там же, где маскируется фон. На набор таких мест единицы.

- **Пересчёт альфы из цвета поверх чистого RGBA.** До 2026-06-11 основной путь
  заливал прозрачные области белым и вычислял прозрачность заново — это СТИРАЛО
  ручную очистку пользователя (почищенная полость лука заливалась белым). Чинится
  путём А (уважать исходную альфу). Старый путь оставлен только для сырых белых листов.

## Известные слабые места

- **Сортировка нарезанных иконок по строкам** (`bboxes.sort(key=b[1]//90)`)
  — фиксированная полоса 90 px ненадёжна на крупном листе: предметы одной строки с
  чуть разным верхним краем попадают в разные полосы, порядок едет. **Чинить ДО
  моста** реестр↔конвейер (мост держится на «N-я иконка = N-й ключ»).

- **Замкнутые окна — вручную.** Раз `punch_internal_white` отключён, петля арбалета
  и дырки колец НЕ становятся прозрачными сами. Их нужно очистить в редакторе в
  исходном листе (путь А их уважит).

- **Бахрома по краям.** На тёмном UI у краёв возможна светлая кайма от антиалиасинга.
  Снимается опциональным `trim_edges` (меню [2] внешний / [3] внутренний) или чистится
  в исходнике. Для поиска проблемных мест — анализатор `analyze.py`.

- **Белые пятна неотличимы от бликов кодом.** 2026-06-11 замером доказано: настоящий
  остаток фона и яркий блик стали по яркости/размеру не различить (оба 245–255). Поэтому
  `analyze.py` белые пятна только ПОКАЗЫВАЕТ («проверь глазами»), а не авто-чинит —
  иначе сотрёт блики. Сами пятна правятся в исходнике.

- **WebP через Pillow, не OpenCV** (часть сборок cv2 теряет альфу при записи WebP).
